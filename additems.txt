Authentication in this project (detailed)
========================================

Overview (what provides auth)
------------------------------
Authentication is handled by **Supabase Auth** using:
- A **browser Supabase client** for sign-in/sign-up/reset flows (`lib/supabase/client.ts`)
- A **server Supabase client** for server rendering and cookie-backed session checks (`lib/supabase/server.ts`)
- A **Next.js middleware** that protects routes, redirects unauthenticated users to `/auth`, and exchanges `?code=` for a Supabase session (`middleware.ts`)

The app also uses “portal roles” (buyer / vendor / services / admin) driven by:
- URL query params on `/auth` (`role` + `next`)
- Supabase `user.app_metadata.role` / `user.app_metadata.roles` (admin checks)
- LocalStorage keys that the app sets after login to personalize each portal UI (e.g. `currentBuyerName`, `currentVendorName`, etc.)

Environment variables
----------------------
Supabase config is read from environment variables (`lib/supabase/env.ts`):
- `NEXT_PUBLIC_SUPABASE_URL`
- `NEXT_PUBLIC_SUPABASE_ANON_KEY`
- `SUPABASE_SERVICE_ROLE_KEY` (server-only, used for admin RPC / admin operations)

Client vs server Supabase clients
----------------------------------
Browser client (`lib/supabase/client.ts`)
- Uses `createBrowserClient(url, anonKey)` from `@supabase/ssr`
- Memoized as a singleton so components reuse one instance

Server client (`lib/supabase/server.ts`)
- Uses `createServerClient(url, anonKey, { cookies })` from `@supabase/ssr`
- Reads/writes auth cookies via Next’s `cookies()` store
- `setAll` is wrapped in try/catch because server components may not allow setting cookies

Route protection (middleware)
------------------------------
The middleware (`middleware.ts`) is the main **gatekeeper** for protected areas.

Protected route prefixes:
- `/buyer/*`, `/vendor/*`, `/services/*`
- Admin: `/admin/*`

Important exception:
- Buyer roadside/help services are intentionally public:
  - `/buyer/services`
  - `/buyer/services/*`

What the middleware does:
1. Creates a Supabase server client wired to request/response cookies.
2. Handles auth callback `?code=...`:
   - Calls `supabase.auth.exchangeCodeForSession(code)`
   - If successful, redirects to the same URL **without** the `code` param
   - Forwards `Set-Cookie` headers so the session cookie persists.
3. Checks current session user: `supabase.auth.getUser()`.
4. If **not signed in** and path is protected:
   - Redirects to `/auth`
   - Adds `next=<original pathname + search>` (so the app can return you where you came from)
   - Adds `role=buyer|vendor|services` based on the path prefix.
5. Admin routes:
   - If not signed in: redirect to `/auth?role=admin&next=...`
   - If signed in but user is not admin (by `app_metadata.role/roles`): redirect to
     `/auth?role=admin&next=/admin&error=admin_required`

Middleware matching:
- Applies broadly, excluding Next static/image routes and common image files via `config.matcher`.

The `/auth` page (multi-portal sign-in / sign-up)
--------------------------------------------------
Main auth UI is `app/auth/page.tsx` and is a client component.

Inputs (from URL):
- `role = searchParams.get('role') || 'buyer'`
- `nextPath = searchParams.get('next') || getDefaultNext(role)`

Default portal landing paths (`getDefaultNext`):
- buyer -> `/buyer`
- vendor -> `/vendor`
- services -> `/services/orders`
- admin -> `/admin`

Session conflict handling (“Already signed in”)
-----------------------------------------------
On load (and whenever `role` changes), `/auth` calls `supabase.auth.getUser()`:
- If there is NO user: show the sign-in form.
- If there IS a user: determine which portal the current session corresponds to:
  - Admin: based on `user.app_metadata.role/roles`
  - Non-admin: based on localStorage hints:
    - `currentVendorName` => vendor
    - `currentServiceProviderName` => services
    - `currentBuyerName` => buyer
- If the session matches the requested portal, it redirects straight to `nextPath`.
- If it’s a different portal, it renders a conflict UI:
  - Button to **sign out and continue** (clears portal localStorage keys and calls `supabase.auth.signOut()`)
  - Optional button to **stay** in the current portal dashboard

Signing in (email + password)
------------------------------
Primary sign-in call:
- `supabase.auth.signInWithPassword({ email, password })`

Post-sign-in checks:
- Admin portal: calls `hasAdminAccess()` (reads `app_metadata`) and signs out if not allowed.
- Buyer portal: applies a phone-number gate (details below).

If checks pass:
- `persistSessionProfile()` writes portal identity data into localStorage
- Navigates using `router.replace(nextPath)`

Sign-up behavior (only after “invalid credentials”)
---------------------------------------------------
If sign-in fails with “invalid login credentials” (and the portal isn’t admin), `/auth` may attempt sign-up.

Before signing up it calls:
- `POST /api/auth/check-email-registered` with `{ email, portal: role }`

That endpoint (`app/api/auth/check-email-registered/route.ts`):
- Uses a service role Supabase client (`lib/supabase/admin.ts`)
- Calls RPC: `auth_portal_signup_blocked(check_email, portal)`
- Returns `{ registered: boolean }`

Then it calls Supabase sign-up:
- Buyer:
  - `supabase.auth.signUp({ email, password, options: { emailRedirectTo: <origin>/auth?role=buyer&next=<nextPath> } })`
  - The redirect ensures that after email verification, Supabase sends the user back into `/auth` with the correct portal and destination.
- Vendor/services:
  - `supabase.auth.signUp({ email, password })` (no explicit redirect configured in this file)

After sign-up:
- If Supabase returns **no session** (common when email confirmation is required), the UI shows a “check your email” success message.
- If a session exists immediately:
  - Buyer flow ensures a `customers` row exists via `/api/customers` (see below)
  - Buyer phone gate may trigger
  - Then it persists localStorage portal identity and redirects to `nextPath`.

Buyer phone gate (required for buyers)
---------------------------------------
Buyers must have a usable phone number so service providers can contact them.

When it checks:
- `gateBuyerPhoneIfNeeded(email)` calls `/api/customers?email=...`
- It considers the phone valid only if it has **at least 9 digits** (digits-only count)

If the phone is missing/too short:
- `/auth` switches to a dedicated “Add your mobile number” step
- User enters a phone number and clicks “Save and continue”

Saving the phone:
1. Reads current user from Supabase (`supabase.auth.getUser()`)
2. Updates or creates the customer record:
   - If customer exists: `PUT /api/customers/:id` with `{ phone }`
   - If not: `POST /api/customers` with `{ id: user.id, name, email, phone, ... }`
3. Calls `persistSessionProfile()` (writes localStorage keys)
4. Redirects to `nextPath`

Persisting portal identity (localStorage)
------------------------------------------
After successful auth, the app stores portal identity hints in localStorage so sidebars/dashboards can display names and IDs.

In `persistSessionProfile()`:
- Vendor portal:
  - `currentVendorId = user.id`
  - `currentVendorName = "<emailPrefix> (vendor)"`
  - Best-effort `POST /api/vendor/bootstrap`
- Services portal:
  - `currentServiceProviderName = "<emailPrefix> (service provider)"`
  - Also sets `currentVendorId` and `currentVendorName` to the same id/name
  - Best-effort `POST /api/vendor/bootstrap`
- Buyer portal:
  - `currentBuyerName = "<emailPrefix> (buyer)"`
  - `currentBuyerEmail = user.email`
  - Fetches `/api/customers?email=...` to store `currentBuyerId` and `currentBuyerPhone` if present

Password reset
---------------
From `/auth` (Forgot password mode):
- Calls `supabase.auth.resetPasswordForEmail(email, { redirectTo: <origin>/auth/reset-password })`

Reset password page (`app/auth/reset-password/page.tsx`):
- Waits and checks `supabase.auth.getSession()`; if no session appears it shows a hint about expired links / redirect URL misconfiguration
- Updates the password using `supabase.auth.updateUser({ password })`
- Redirects back to `/auth`

End-to-end flow summary
------------------------
1. User visits a protected route (e.g. `/buyer/orders`).
2. Middleware sees no Supabase user -> redirects to `/auth?role=buyer&next=/buyer/orders`.
3. User signs in (or signs up) via Supabase on `/auth`.
4. Buyer-only: may be required to add a phone number before continuing.
5. `/auth` persists portal identity hints in localStorage and navigates to `nextPath` with `router.replace`.
6. If Supabase returns the user via a `?code=` callback, middleware exchanges it for a session and cleans the URL.
