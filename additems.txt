Header profile icon (small screens) — implementation & behavior
===============================================================

Where it lives (mobile-only header)
-----------------------------------
The small-screen header is rendered inside `components/header.tsx` under a wrapper with `className="md:hidden"`.
In that mobile header row, the profile control sits to the right of the cart button.

DOM structure (mobile)
----------------------
The profile UI is a `div` wrapper with:
- `ref={mobileProfileMenuRef}`
- `className="relative"`
- `onMouseEnter={handleProfileHoverOpen}`
- `onMouseLeave={scheduleProfileHoverClose}`

Inside it:
1. A **button** that shows the profile icon (`UserCircle2`) + chevron (`ChevronDown`)
   - `aria-label="Open profile menu"`
   - `aria-expanded={profileMenuOpen}`
   - `onClick={toggleProfileMenu}`
2. A conditional dropdown panel rendered only when `profileMenuOpen` is true:
   - `div` positioned `absolute right-0 mt-2 w-56 ... z-50`

Open/close state
----------------
The dropdown visibility is controlled by shared state in the header component:
- `const [profileMenuOpen, setProfileMenuOpen] = useState(false)`
- `const toggleProfileMenu = () => setProfileMenuOpen(prev => !prev)`

Closing when tapping outside
----------------------------
`components/header.tsx` attaches a document-level `mousedown` handler:
- It checks whether the click target is inside either the mobile menu (`mobileProfileMenuRef`) or the desktop menu (`desktopProfileMenuRef`).
- If the click is outside both, it runs `setProfileMenuOpen(false)`.

This is the primary “tap outside to close” behavior on small screens.

Hover behavior (also wired on mobile)
-------------------------------------
Even though small screens are primarily tap-driven, the mobile wrapper still wires hover handlers:
- `handleProfileHoverOpen()` immediately opens the menu (`setProfileMenuOpen(true)`)
- `scheduleProfileHoverClose()` closes it after ~150ms and resets the menu mode back to `'login'`
  - It uses `profileCloseTimerRef` to debounce/avoid flicker.

Menu content and modes (login vs join)
--------------------------------------
The dropdown has two internal “modes” controlled by:
- `profileMenuMode` state (`'login' | 'join'`)

When `profileMenuMode === 'login'`, it shows **login links**:
- `Login as Buyer` -> `/auth?role=buyer&next=/buyer`
- `Login as Vendor` -> `/auth?role=vendor&next=/vendor`
- `Login as Service Provider` -> `/auth?role=services&next=/services/orders`
Plus a “Join MyGarage” button which flips the mode to `'join'`.

When `profileMenuMode === 'join'`, it shows **join links** (same destinations, but labeled Join):
- `Join as Buyer` -> `/auth?role=buyer&next=/buyer`
- `Join as Vendor` -> `/auth?role=vendor&next=/vendor`
- `Join as Service Provider` -> `/auth?role=services&next=/services/orders`
Plus a “My Account” button which flips back to `'login'`.

What “landing page” you get after tapping an item
--------------------------------------------------
All menu items route to the unified auth page (`/auth`) with:
- `role=<buyer|vendor|services>` to choose the portal
- `next=<portal root>` so, after a successful login, `/auth` redirects using `router.replace(next)`

In short: on small screens, tapping the profile icon toggles an absolute-positioned dropdown; tapping outside closes it; choosing an option navigates to `/auth?...` to sign in/join and then land on the relevant dashboard.
